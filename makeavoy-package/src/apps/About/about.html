Petrichor64

An ambitious project intended to create a retro game engine/fantasy console like the 5th generation of game consoles. Back when 3D was fresh and polygonal and sound synthesis was cheap and limited. The constraints similar to fantasy consoles like PICO-8 or TIC-80 offer developers restricted sandbox environment which forces creative solutions to implement desired game projects. Although not as limited due to using a full 3D renderer built using the WGPU graphics API, the goal is for the Lua scripting layer to allow anyone to quickly make interesting 3D games and applications that run anywhere

In addition to cross platform and music synthesis, the system features easy asset loading, 3D tiling system that doubles as a 2D system if desired, fast drawing rasters that can be used as textures for sprites or 3d models. Models can be built inline via the mod command, or loaded in as gltf/glb files. The full CLI in-engine allows quick building, loading, and bundling game projects

This project is ongoing since early 2019 and has many features still in the work. Currently a lua interpreter is being built from scratch so as to allow for compiling to wasm32-unknown-unknown targets, and this make the engine web-compatible

Makeavoy

This site, Makeavoy.com, utilizes all of my favorite techniques I've picked up over the years. It's a little experimental as it's exploring what's possible for a single page application, so don't be surprised if the source is a little bit sloppy! There's a lot going on under the hood so I'm going to break it down here...

Backend

Originally using node.js with vhost in express.js to allow multiple domains on the same server, it's since been moved over to a rust built axum/tower server that serves either MakeAvoy.com or Petrichor64.app depending on the hostname provided. Socket.io was originally used for WebSockets and currently a rust alternative is WIP. There's also MariaDB and SQLite3 databases for testing user authentication. After that, various projects are separated by folders and sub modules to keep things organized. It's all rolled out on my locally hosted Ubuntu server.

FrontEnd

The main site utilizes a lot of 3D via Three.js. Due to the above average script sizes, everything is lazy loaded on demand in order to reduce the Time to Interaction. The Rendering system is triggered to load at runtime, while each additional "app" is loaded upon clicking so as to only load the necessary content, at the cost of some load time. The initial payload hosts the basics such as the 3D CSS loading animation, and the app navigation system. The site loads in a self made SVG title that 'draws' into place by a stroke-dashoffset. It also includes dynamic app button movement and placement a la an iPad, and a snazzy vector based drawing system to animate the navigation bar using perpendicular velocity offsets. The title accommodates for the bar as well. Finally, each app when loaded in applies to Canvas3D the scene contents and animation cycles. Upon swapping apps, the buffer is dumped to an image raster and stealthily swapped in. This was done for an old Three.js limitation but is no longer necessary.All 3D and SVG assets are my own.

Site was originally pure vanilla JS, You can view this here but was recently converted to a typescript and webpack 5 setup for better upkeep

Petrichor64 Command Site

A Svelte built site utilizing markdown dumped from an automated build process on every release of Petrichor64. Markdown is left inline under command implementations via rust macros and composited into a single file and uploaded to server. Site also utilizes a command tag based search bar upon typing or using the search button on mobile.

Code is hosted as a mono repo with backend and makeavoy.com, as they share a common ground

Social Game - experimental online board game

This was an extremely ambitious project. As mentioned before, the server hosts an instance of Cannon.js that keeps synced up with local user instances of Cannon.js in order to replicate a multiplayer physics sandbox experience on a hexagonal grid game platform. The current results are less then ideal as the updates are not constant in order to not strain the server. But aside the occasional physics artifacts it has the ability to 'store' 3D items into the DOM, send messages including 16x16 images, modify the terrain on the fly, and user authentication and administration. There was also a D&D character sheet implemented that saves into a sqlite3 database. Everything is very WIP. An "infinite" terrain system is in the works, as well as height maps, inverse kinematics animation systems, better environmental GLSL effects, and waypointing. Like i said, extremely ambitious. But I'm super happy with it so far!

Desk.js - User Interface

This experimental interface is intended for touchscreen. Created as means to test the possibility of managing multiple window or objects with separate hands and fingers, It registers all of 10 fingers separately. You can drag an item with one finger while moving another item with a separate finger, or resize one item with one hand while moving or expanding another interdependently. When items overlap, they can be moved together just like you could with a stack of papers (toggleable logic). You can also drag an image file to it from the desktop to paste in images, or double tap a square to paste text. Context menu (long press or right click) the desktop brings up a radial menu. The entire project is a bit rough and really needs an update to iron out the bugs, including some lag only on iOS.

Reserver

This was another simple PWA that evolved for work necessity. 12 members of our team were forced to use a single citrix portal in order to management client incidents. Anytime someone used it, if someone else was working with it already, they would get kicked out. We did our best to alert each other but sometimes we would miss messages. I created this app as a stopgap. It's a simple neomrophic button with gradient background opacity animations (since we still can't tween background gradients...) that sends a socket.io WebsScket message to the server reserving your status and alerting all connected clients of the state change. Non occupying clients can then "pend" the user sending them desktop notifications (if they opted in) and even posting in our discord chat via webhooks that they were waiting. Full PWA manifest and desktop icons included for pinning to the taskbar!

Staddy

This project was a bit sloppy, but it was a necessary step in order to organize my tech support job. There were some existing options out there but nothing quote met my needs, so I made it instead. It's effectively an advanced bookmark manager, with note taking, and ideally password management. What started as a joke became crucial for organizing my workflow, as we supported over 40 unique clients' stacks and a couple hundred sub clients. As I couldn't safely store passwords, this aspect got scraped as I was unable to meet the job's encryption requirements at the time. As all clients can be added in manually and stored as a JSON object through localstorage, it was infinitely customizable and exportable. It stored unique notes for each client as well, and even stored background color preferences. There was a break system being incorporated via socket.io but ultimately fell to the wayside as well. I hope to clean up this logic for some sort of organizational app someday.
